# PharmaNavi セキュリティ設計書

開発時に必ず遵守するセキュリティ要件一覧。実装のたびに本ドキュメントを参照すること。

---

## 1. 認証・アクセス制御

### 1.1 認証方式
| 項目 | 要件 | 実装先 |
|------|------|--------|
| ログイン | メールアドレス + パスワード | Supabase Auth |
| 二要素認証 | メール認証コード（ログイン時） | Supabase Auth 2FA |
| パスワード強度 | 8文字以上、英数字含む | Supabase Auth 設定 |
| セッション | JWT、有効期限管理 | Supabase デフォルト |

### 1.2 権限設計（RBAC）
| ロール | できること | できないこと |
|--------|------------|-------------|
| 管理者 | 全機能、ユーザー管理、加算マスタ、全店舗 | - |
| 店舗管理者 | 自店舗＋担当店舗のデータ、スタッフ管理 | 他店舗の編集、マスタ変更 |
| 一般ユーザー | 自分のアクションプラン、進捗確認 | 他ユーザーの編集、CSV取込 |
| データ入力者 | CSV取込、手動入力 | ダッシュボード閲覧制限あり |

### 1.3 未認証時の扱い
- ログイン画面以外はすべて認証必須
- 未ログインで保護ページにアクセス → `/login` にリダイレクト
- APIルートはすべてセッション検証

## 2. データベース（Supabase）

### 2.1 Row Level Security（RLS）
**全テーブルでRLSを有効にすること。例外なし。**

| テーブル | RLSポリシー方針 |
|----------|-----------------|
| pharmacies | 自組織の店舗のみ閲覧・編集 |
| users | 管理者は全件、店舗管理者は自店舗のユーザーのみ |
| monthly_records | 自組織の店舗のデータのみ |
| kasan_status | 同上 |
| action_tasks | 自分のタスク or 管理対象のスタッフのタスク |
| csv_import_logs | 自組織の店舗のログのみ |
| kasan_master | 全員閲覧可、編集は管理者のみ |

### 2.2 組織・店舗の分離
- `organization_id` または `pharmacy_id` でテナント分離
- クエリに必ず `WHERE pharmacy_id IN (ユーザーがアクセス可能な店舗)` を含める
- RLSで二重チェック

### 2.3 個人情報の扱い
- **患者氏名・生年月日は一切取り込まない**（要件定義書準拠）
- レセコンCSV取り込み時、個人を特定できる列は破棄する
- 集計値・件数のみを `monthly_records` に保存

## 3. API・通信

### 3.1 通信暗号化
- HTTPS（TLS 1.2以上）必須
- Vercel・Supabase はデフォルトでHTTPS

### 3.2 APIルートの保護
```javascript
// 全APIで実施すること
// 1. セッション検証
const { data: { session } } = await supabase.auth.getSession();
if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

// 2. 権限チェック（必要に応じて）
// 3. 入力値の検証（後述）
```

### 3.3 Claude API呼び出し
- APIキーは**サーバーサイドのみ**（`/api/route.js` から呼ぶ）
- フロントエンドに `ANTHROPIC_API_KEY` を絶対に書かない
- 環境変数は Vercel の Environment Variables で管理

## 4. 入力検証・サニタイズ

### 4.1 必須チェック
- 必須項目の空チェック
- 型チェック（数値、日付、メール形式）

### 4.2 CSV取り込み時
- ファイルサイズ制限（例：10MB以下）
- 行数制限（例：10万行以下）
- 拡張子チェック（.csv のみ許可）
- 文字コード：UTF-8 を想定、BOM付きにも対応

### 4.3 手動入力時
- 数値範囲チェック（負数不可、妥当な上限）
- 日付形式チェック（YYYY-MM）
- SQLインジェクション対策：Supabase クライアントはパラメータ化済み

### 4.4 XSS対策
- React はデフォルトでエスケープ
- `dangerouslySetInnerHTML` は使用禁止（やむを得ない場合はサニタイズ必須）
- ユーザー入力値をそのままHTMLに出力しない

## 5. 環境変数・シークレット

### 5.1 管理ルール
| 変数 | 公開場所 | 秘匿場所 |
|------|----------|----------|
| NEXT_PUBLIC_* | フロントに公開される | 使用禁止（Supabase URL等のみ） |
| SUPABASE_SERVICE_ROLE_KEY | 絶対にフロントに出さない | サーバー・APIのみ |
| ANTHROPIC_API_KEY | 同上 | 同上 |

### 5.2 .env.local
- `.env.local` は `.gitignore` に含める（必ず確認）
- 本番のキーをローカルにコピーしない

---

## 6. 監査・ログ

### 6.1 記録するもの
| イベント | 記録内容 |
|----------|----------|
| CSV取り込み | 日時、ユーザーID、店舗、ファイル名、件数 |
| ログイン | Supabase Auth のログを利用 |
| 加算マスタ変更 | 管理者による変更履歴 |
| 手動入力 | 誰がいつどの店舗のデータを更新したか |

### 6.2 ログの保持
- `csv_import_logs` テーブルで取り込み履歴を保持
- 改定データ取り込み時も同様にログを残す

## 7. デプロイ・本番環境

### 7.1 ビルド前チェック
- [ ] `npm run build` が通る
- [ ] 環境変数がVercelに設定済み
- [ ] RLSが全テーブルで有効
- [ ] 患者個人情報を扱うコードがない

### 7.2 Vercel設定
- 本番・プレビュー・開発で環境変数を分離
- 本番のみ厳格なCORS設定（必要に応じて）
