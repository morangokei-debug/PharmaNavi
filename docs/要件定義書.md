# PharmaNavi 要件定義書

**版数**: 1.1  
**作成日**: 2026年3月  
**更新日**: 2026年3月（令和8年度改定対応）  
**対象**: 薬局オーナー向け経営支援Webアプリケーション

---

## 1. プロジェクト概要

### 1.1 目的

薬局オーナー（非エンジニア）が一人で利用する経営支援Webアプリケーションを提供する。調剤報酬改定に伴う加算の取得状況を可視化し、達成に向けたロードマップとAIアドバイスにより経営判断を支援する。

### 1.2 対象ユーザー

- **主ユーザー**: 薬局オーナー（経営者）
- **想定規模**: 1組織・複数店舗まで
- **前提**: プログラミングや開発環境に関する専門知識は持たない

### 1.3 スコープ

- 令和8年度（2026年）診療報酬改定の主要加算を対象（2026年6月1日施行）
- 月次実績データの入力・集計・可視化
- 加算達成状況の自動判定とロードマップ表示
- AI（Gemini API）による経営アドバイス

---

## 2. システム概要

### 2.1 アーキテクチャ

| 項目 | 内容 |
|------|------|
| フロントエンド | Next.js 14（App Router）、React 18、Tailwind CSS |
| バックエンド | Next.js API Routes（サーバーサイド） |
| データベース | Supabase（PostgreSQL） |
| 認証 | Supabase Auth |
| AI | Google Gemini API（サーバーサイドのみ） |
| デプロイ | Vercel |
| グラフ | Recharts |

### 2.2 セキュリティ方針

- Claude/Gemini APIキーはフロントエンドに露出させない（必ずAPI Route経由で呼び出す）
- 患者データを含むテーブルにはRLS（Row Level Security）を設定
- 組織単位でデータを分離（同一組織内のユーザーのみアクセス可能）

## 3. 機能要件

### 3.1 認証・ユーザー管理

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| AUTH-01 | ログイン | メールアドレス・パスワードによるログイン |
| AUTH-02 | ログアウト | セッション終了 |
| AUTH-03 | プロファイル自動作成 | 初回サインアップ時に pharma_profiles を自動作成 |
| AUTH-04 | 組織紐付け | プロファイルに organization_id を紐付け、以降のデータアクセスを組織単位に制限 |

### 3.2 設定（組織・店舗・届出）

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| SET-01 | 組織作成 | 組織名を入力して組織を作成し、自プロファイルに紐付け |
| SET-02 | 組織編集・削除 | 組織名の変更、組織の削除（関連店舗・データも連鎖削除） |
| SET-03 | 店舗登録 | 組織に紐づく店舗を追加（店舗名、住所、許可証番号） |
| SET-04 | 店舗編集・削除 | 店舗情報の変更、店舗の削除 |
| SET-05 | 届出管理 | 店舗ごとに届出状況をチェック（地域支援・医薬品供給対応体制加算、在宅患者訪問薬剤管理指導など） |
| SET-07 | 調剤基本料区分 | 店舗ごとに調剤基本料１・２・３を設定（地域支援加算2〜5の判定に使用） |
| SET-06 | 組織未設定時の案内 | 組織・店舗が未設定の場合、設定画面への誘導メッセージを表示 |

### 3.3 データ入力

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| INPUT-01 | 月次実績の手動入力 | 過去12ヶ月分の月次実績を手動で入力（店舗選択、年月ごとの入力欄） |
| INPUT-02 | 入力項目 | 処方箋受付件数、在宅訪問件数、服薬フォロー、かかりつけ算定、麻薬調剤、残薬調整・有害事象防止、マイナ確認割合、電子処方箋、時間外対応、外来服薬支援、服薬情報提供、小児特定、多職種連携会議出席、後発医薬品調剤割合など |
| INPUT-03 | CSV一括取り込み | CSVファイルをアップロードし、列名を自動マッピングして月次実績に合算 |
| INPUT-04 | 取り込み先月の指定 | CSV取り込み時に対象月を選択可能（デフォルトは当月） |
| INPUT-05 | 保存時の加算再計算 | 保存時に pharma_kasan_status を自動再計算（API経由） |
| INPUT-06 | 店舗選択の永続化 | 選択した店舗を localStorage に保存し、リロード後も復元 |

### 3.4 加算ロードマップ

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| ROAD-01 | 12ヶ月推移表示 | 各加算について過去12ヶ月の達成状況をタイムラインで表示 |
| ROAD-02 | 達成状況の色分け | 達成（緑）、一部達成（黄）、未達（グレー）で視覚化 |
| ROAD-03 | 条件別進捗バー | 届出状況・実績値の達成度をプログレスバーで表示 |
| ROAD-04 | 店舗切り替え | 複数店舗がある場合、プルダウンで切り替え（選択を永続化） |
| ROAD-05 | 加算マスタ連携 | pharma_kasan_master の requirements_json に基づき条件を表示 |
| ROAD-06 | 取得に向けた不足分 | 未達成加算について「あと何をどれくらいで取得できるか」を表示 |

### 3.5 管理者画面

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| ADMIN-01 | 複数店舗一覧 | 組織内の全店舗の加算取得状況を表形式で一覧表示 |
| ADMIN-02 | 12ヶ月タイムライン | 各店舗・各加算の過去12ヶ月の達成状況を色分けで表示 |

### 3.6 ダッシュボード

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| DASH-01 | サマリー表示 | 算定済み加算数、達成率、未達加算数をカード表示 |
| DASH-02 | グラフ表示 | 加算達成状況の推移をグラフで表示（Recharts） |
| DASH-03 | 初回案内 | 組織・店舗未設定時は設定画面、データ未入力時はデータ入力画面への誘導 |

### 3.7 AIアドバイス

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| AI-01 | チャット形式の相談 | ユーザーが質問を入力し、AIが加算・経営に関するアドバイスを返答 |
| AI-02 | サジェスト | よくある質問をボタンで選択可能 |
| AI-03 | サーバーサイド呼び出し | Gemini APIは /api/chat 経由でサーバーサイドのみから呼び出す |

### 3.8 タスク管理

| 機能ID | 機能名 | 説明 |
|--------|--------|------|
| TASK-01 | タスク登録 | 加算達成に向けたタスクを登録（タスク内容、期限、関連加算） |
| TASK-02 | タスク一覧 | ステータス（未着手・進行中・完了）でフィルタ表示 |
| TASK-03 | タスク更新 | ステータス変更、編集、削除 |
| TASK-04 | ユーザー紐付け | タスクはログインユーザーに紐づく |

## 6. 非機能要件

### 6.1 パフォーマンス

- ページ初回表示: 3秒以内を目安
- データ入力・保存: レスポンス 2秒以内を目安

### 6.2 可用性

- Vercel の SLA に準拠
- Supabase の無料枠または有料プランの SLA に準拠

### 6.3 運用

- デプロイ前に `npm run build` を必ず実行し、ビルドエラーがないことを確認
- 環境変数（APIキー、Supabase URL/Key）は Vercel の環境変数で管理

### 6.4 制約事項

- 一度に1つの変更のみ実施し、動作確認後に次へ進む
- 複数ファイルの同時大幅変更を避ける
- 患者データを含むテーブルには RLS を設定する
